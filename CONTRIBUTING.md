# Contributing

A note on expectations first: this repository is the source code of my personal website and shop. It is open source primarily for transparency — so anyone can read exactly what the shop collects about its users and how it tracks them (it doesn't) — not because it is a community project with a roadmap. I don't expect contributions.

That said, issues and pull requests are genuinely welcome if you've found a bug or want to improve something. Just keep in mind that the code follows the needs of my own site: for anything larger than a fix, please open an issue first, and features that don't fit what I run may be better kept on your fork. Everything below gets you to a working development setup.

## Development Setup

### Prerequisites

- Go 1.26+ (no CGO required)
- Node.js 20+ and npm
- Docker and Docker Compose (for local dev stack — includes Postgres)
- [just](https://github.com/casey/just) command runner (optional but recommended)

### Quick Start

```bash
# Clone the repo
git clone https://github.com/SteVio89/stevio-home.git
cd stevio-home

# Start the local dev stack (backend + frontend + Postgres)
just up

# Or without just:
docker compose -f docker-compose.dev.yml up --build -d
```

The dev stack runs:
- **Postgres** on `:5432` (volume-persisted; `just clean` to wipe)
- **Backend** on `:8080` (Go API server)
- **Frontend** on `:3000` (nginx serving the React build, proxying `/api/`, `/auth/`, `/sitemap.xml`, `/robots.txt` to backend)

For frontend hot-reload during development:

```bash
cd frontend && npm install && npm run dev
```

This starts Vite on `:5173` with a proxy to `:8080` for API requests.

In dev mode without SMTP configured, magic-link emails log to backend stdout. Grep for the verify URL:

```bash
docker compose -f docker-compose.dev.yml logs backend | grep -A12 "mail: not configured"
```

### Running Tests

Backend integration tests need a Postgres instance. They skip cleanly when `TEST_DATABASE_URL` is unset, so a bare `go test ./...` passes for compile-only validation.

```bash
# Compile-only (no DB needed; integration tests skip)
just test
# or: cd backend && go test ./...

# Full integration tests against the dev Postgres
docker compose -f docker-compose.dev.yml up -d postgres
TEST_DATABASE_URL="postgres://stevio:stevio-dev-password@localhost:5432/stevio?sslmode=disable" \
  cd backend && go test ./...

# Frontend tests
just test-frontend
# or: cd frontend && npx vitest run

# All tests
just test-all

# Go formatting, build, tests, vet, and lint in one go
just test
```

Per-test isolation: each `testutil.SetupTestDB(...)` call creates a fresh schema named `test_<random>` and pins `search_path` to it. The schema is dropped on `t.Cleanup`. Tests run in parallel safely.

### Generating Secrets

```bash
# Generate signing key secret (encrypts Ed25519 keys at rest)
just gensigningsecret

# Generate session secret (64-char hex)
just gensecret

# Generate email hash salt (64-char hex)
just gensalt
```

## Code Style

### Backend (Go)

- stdlib `net/http` router — no external web framework
- Handlers are methods on `*handlers.Handlers` (dependency injection)
- DB queries use `database/sql` with Postgres `$1, $2, ...` placeholders (no ORM)
- Postgres dialect: `BOOLEAN` (not INTEGER 0/1), `TIMESTAMPTZ` (not TEXT), `NOW()` (not strftime), `ON CONFLICT` (not OR IGNORE)
- For dynamic WHERE clauses, build with `?` placeholders and call `dbutil.NumberPlaceholders(sql, 0)` to convert to `$N` (see `db/queries/orders.go ListAllOrders` for the pattern)
- For IN-clause placeholders, use `dbutil.InPlaceholders(count, offset)`
- Error responses use `apierr.Write(w, apierr.ErrSomething)` sentinel errors
- Constraint-violation detection: check `pgconn.PgError.Code == "23505"` for unique violations — never string-match the error message

### Data Model Conventions

- **Projects** own display content (slug, title, tagline, description, image, gallery, `has_detail_page`)
- **Apps** are an optional 1:1 commerce attachment (`apps.project_id`) carrying bundle id, price, versions, system requirements
- Translations: `entity_type='project'` for title/tagline/description, `entity_type='app'` only for `system_requirements`, `entity_type='project_image'` for alt text, `entity_type='version'` for release notes
- IDs are TEXT primary keys generated by `dbutil.NewID()` (UUID v4)

### Frontend (React/TypeScript)

- Single global `index.css` with CSS custom properties — no inline styles, no CSS modules, no Tailwind
- Toast notifications via `useToast().addToast()` — never `alert()`
- Auth state from `AuthContext`, site config from `SiteConfigContext`
- All prices displayed via `formatPrice(cents, currencySymbol)`
- API types live in `src/api/client.ts`; mirror Go struct field names exactly
- i18n key namespaces: `project.*` for content, `commerce.*` for pricing/buy buttons, `app.*` reserved for cross-domain UI

## Pull Request Guidelines

1. For anything beyond a small fix, open an issue first so we can agree it belongs here rather than on a fork
2. Fork the repo and create a feature branch from `main`
3. Make your changes with clear, focused commits
4. Ensure all tests pass (`just test-all` — including `TEST_DATABASE_URL` for integration coverage)
5. Ensure the build succeeds (`just build`)
6. Open a PR with a clear description of what and why

## Reporting Issues

Please use [GitHub Issues](https://github.com/SteVio89/stevio-home/issues) to report bugs or suggest improvements. Include:

- Steps to reproduce (for bugs)
- Expected vs actual behavior
- Environment details (OS, Go version, browser)

For security vulnerabilities, do not open a public issue — see [SECURITY.md](SECURITY.md).
